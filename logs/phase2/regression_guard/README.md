# logs/phase2/regression_guard

<!-- RCC-MINI-README:START -->

## Purpose

Generated Phase 2 regression-guard artifacts for locking the current evidence baseline before any controlled classifier behavior change.

## S - Formal specification

This folder stores guard reports generated by `scripts/run_phase2_regression_guard.py`. The guard layer reads benchmark, attribution, and candidate-selection artifacts and emits a baseline lock plus acceptance thresholds for future classifier deltas.

## H - Hooks and integration edges

- `hlinet/eval/regression_guard.py` implements regression-guard generation.
- `scripts/run_phase2_regression_guard.py` is the human-facing runner.
- Reads `logs/phase2/benchmarks/latest_phase2_benchmark.json`.
- Reads `logs/phase2/attribution/latest_phase2_attribution.json`.
- Reads `logs/phase2/candidates/latest_phase2_candidate_plan.json`.
- Outputs inform future Phase 2.6B controlled classifier deltas.

## A - Artifacts

- `latest_phase2_regression_guard.json`
- `latest_phase2_regression_guard.md`

## T - Theory or method basis

Regression guards protect the current evidence baseline before classifier behavior changes. The goal is to prevent blind improvements that reduce top-3, destroy HL-unique wins, worsen victim classes, or move false positives into new attractors.

## I - Invariants

- Regression tooling does not change classifier behavior.
- Regression tooling does not claim accuracy improvement.
- Regression tooling does not prove classifier correctness.
- Validation regression checks are not final test evidence.
- Future classifier changes must rerun diagnostics, benchmarks, attribution, candidate selection, and regression guard.

## E - Example

Run from the repository root:

    python scripts/run_phase2_regression_guard.py
