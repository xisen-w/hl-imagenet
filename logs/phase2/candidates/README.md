# logs/phase2/candidates

<!-- RCC-MINI-README:START -->

## Purpose

Generated Phase 2 candidate-selection artifacts for choosing future classifier-change targets from diagnostics, benchmarks, and sample-level attribution.

## S - Formal specification

This folder stores candidate plans generated by `scripts/run_phase2_candidates.py`. The candidate-selection layer reads existing evidence artifacts and emits ranked candidate sets without changing classifier behavior.

## H - Hooks and integration edges

- `hlinet/eval/candidates.py` implements candidate selection.
- `scripts/run_phase2_candidates.py` is the human-facing runner.
- Reads `logs/phase2/diagnostics/latest_phase2_diagnostic.json`.
- Reads `logs/phase2/benchmarks/latest_phase2_benchmark.json`.
- Reads `logs/phase2/attribution/latest_phase2_attribution.json`.
- Outputs inform future Phase 2.6 controlled classifier deltas.

## A - Artifacts

- `latest_phase2_candidate_plan.json`
- `latest_phase2_candidate_plan.md`

## T - Theory or method basis

Diagnostics show aggregate failure geometry. Benchmarks show baseline comparison. Attribution shows individual sample behavior. Candidate selection decides what deserves inspection before any classifier behavior changes.

## I - Invariants

- Candidate tooling does not change classifier behavior.
- Candidate tooling does not claim accuracy improvement.
- Candidate tooling does not prove classifier correctness.
- Validation candidate selection is not final test evidence.
- Future classifier changes must rerun diagnostics, benchmarks, attribution, and candidate selection.

## E - Example

Run from the repository root:

    python scripts/run_phase2_candidates.py
